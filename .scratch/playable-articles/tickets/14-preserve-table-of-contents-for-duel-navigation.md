# Preserve Wikipedia Table of Contents for Duel Navigation

Status: ready-for-agent
Scope: MVP required
Category: enhancement

## Parent

- [`playable-articles/10`](./10-render-typed-article-documents.md)
- [Playable Articles spec](../spec.md)

## Problem

Quick Lab checks show that a Wikipedia page's table of contents can be included
in the rendered Article Document text. This adds navigation chrome to the
article body and makes it unclear which content is available for the actual
race.

## What to build

Investigate how Wikipedia table-of-contents structures arrive at the
normalizer, remove the table of contents from rendered article body text, and
preserve a server-owned representation that the Duel screen can use as a quick
way to jump to headings in the current article. A table-of-contents jump must
not become a Navigation or count as a race click.

## Acceptance criteria

- [ ] The triage investigation identifies the upstream structures that produce
  the table of contents and the current leakage path.
- [ ] Rendered Article Document body content excludes table-of-contents labels,
  links, and surrounding presentation chrome without removing real article
  content.
- [ ] The retained table of contents has a typed or otherwise explicit contract
  that the Duel screen can consume for same-article heading jumps.
- [ ] The Playable Article Lab renders the retained table of contents in a
  dedicated component, visually paired with the diagnostics panel and placed to
  the left of the rendered Playable Article.
- [ ] A table-of-contents jump does not create a Navigation, change the
  authoritative article path, or increment the race click count.
- [ ] Nested headings, missing or malformed entries, accessibility, and
  headings removed by normalizer policy are covered by the implementation plan
  and regression fixtures.

## Triage note

This is intentionally a planning ticket rather than a settled implementation
design. Triage should decide the representation, ownership between normalizer
and renderer, heading-link identity policy, and how a same-article jump is
delivered to the future Duel screen before moving it to ready-for-agent.

## Agent Brief

**Category:** enhancement
**Summary:** Preserve a server-owned table of contents for same-article jumps while keeping TOC chrome out of rendered article body content.

**Current behavior:**
Playable Articles are built from Wikipedia HTML returned by the gateway and normalized into the typed `ArticleDocument` contract. The normalizer already excludes several unsupported or chrome-like structures, including navboxes, edit controls, references, infoboxes, galleries, and tables. It does not currently model a table of contents, heading jump identity, or a same-article jump action. Because incidental containers are otherwise traversed recursively, upstream table-of-contents containers that are not excluded can be flattened into ordinary Article Document headings, lists, text, or unsupported-link labels. Quick Lab checks have shown this can make Wikipedia navigation chrome appear as article body content.

**Desired behavior:**
The server should recognize Wikipedia table-of-contents structures during normalization. TOC presentation chrome should not appear in `ArticleDocument` body blocks, and links inside TOC chrome should not become `Navigation Node`s or candidate article destinations.

The Article Document contract should retain a typed, server-owned table of contents derived from the headings that survive normalization. Each TOC entry should have a stable same-article target identity that maps to a rendered heading in the current article. Nested heading structure should be represented explicitly enough for clients to render a useful outline without inferring hierarchy from raw text.

The Playable Article Lab should render the retained table of contents through a dedicated component. In the Lab layout, this component should appear in a panel visually paired with the diagnostics panel and positioned to the left of the rendered Playable Article. Activating a TOC entry should jump focus/scroll position to the corresponding heading in the current article. It must not call the Navigation callback, request a new article, mutate Lab navigation history, create a gameplay `Navigation`, change the authoritative article path, or increment any race click count when later reused by Duel UI.

**Key interfaces:**
- `ArticleDocument` should expose the retained table of contents and stable heading jump targets as typed data, not raw HTML or client-parsed upstream markup.
- Heading blocks should carry enough identity for a TOC entry to target the rendered heading accessibly and deterministically.
- The Article Document renderer should render heading identities without changing Navigation Node behavior.
- The Lab should include a reusable table-of-contents component that consumes the typed Article Document TOC contract and performs same-article jumps only.
- Normalizer diagnostics should continue to account for omitted upstream chrome, including TOC structures, without exposing raw HTML.

**Acceptance criteria:**
- [ ] Upstream TOC structures from parsed Wikipedia HTML are covered by focused regression fixtures, including at least the structures observed in the Lab and representative MediaWiki TOC class/id patterns.
- [ ] Rendered Article Document body content excludes TOC labels, numbering, links, wrappers, and surrounding presentation chrome while preserving real article headings, paragraphs, lists, figures, captions, and Navigation Nodes.
- [ ] Links found only inside excluded TOC chrome are not resolved as article candidates and never become `Navigation Node`s.
- [ ] The Article Document contract includes a typed table of contents whose entries target normalized rendered headings in the same article.
- [ ] TOC entries are derived only from headings retained by normalizer policy; malformed entries, missing headings, duplicate heading labels, and skipped heading levels produce deterministic, accessible output.
- [ ] The Lab renders a dedicated table-of-contents component in a diagnostics-like panel to the left of the Playable Article at desktop sizes, with a responsive layout that remains usable on narrow screens.
- [ ] Activating a Lab TOC entry scrolls or focuses the matching heading without sending a preview request, changing Lab navigation history, or invoking the Navigation callback.
- [ ] Focused tests cover server normalization, contract serialization, renderer heading targets, Lab TOC rendering, keyboard/mouse activation, and the non-Navigation behavior of TOC jumps.

**Out of scope:**
- Implementing full production Duel screen layout beyond the typed contract and same-article jump semantics needed for later Duel reuse.
- Treating TOC entries as Navigation Nodes or Wikipedia page destinations.
- Preserving Wikipedia's original TOC styling, numbering behavior, collapsible controls, or raw anchor markup.
