# Render Wikipedia Infoboxes

Status: completed
Scope: MVP optional
Category: enhancement

Completed: 2026-07-12 04:41 PM

Implementation: Added semantic Infobox normalization, shared link/image
discovery, typed media placeholders, production rendering, responsive
presentation, and focused server/client coverage.

## Agent Brief

**Category:** enhancement

**Summary:** Preserve supported Wikipedia Infobox content as a dedicated typed
Article Document block and render it through a reusable production Infobox
component inside the Playable Article area.

**Current behavior:**

Infobox subtrees are deliberately excluded from candidate-link discovery,
candidate-image discovery, Article Document normalization, and rendering. As a
result, identifying media, concise facts, and otherwise eligible routes shown in
Wikipedia Infoboxes are absent from Playable Articles. The baseline Playable
Article pipeline, canonical Navigation classifier, attributed-image policy,
typed renderer, reusable Playable Article area, and development Lab are now in
place, so the original pipeline blocker is cleared.

**Desired behavior:**

Recognize Wikipedia Infoboxes structurally and represent each one as a dedicated
semantic Article Document block rather than as a generic table or upstream HTML.
Retain every supported Infobox item in source order without importance
heuristics or arbitrary item caps. Supported content includes an optional title,
internal section labels, fact rows with optional labels, readable line and
paragraph grouping, simple lists, the existing inline vocabulary, qualifying
images and captions, and typed audio/video placeholders.

Normalization should degrade at the smallest safe boundary. When supported and
unsupported content coexist, retain the supported text, structure, Navigation
Nodes, and images while omitting only the unsupported subtree. Drop an Infobox
item only when nothing meaningful remains. Infobox content supplements the
article body: it must not make an otherwise empty Article Document pass the
existing meaningful-body requirement.

Every eligible internal article link anywhere in retained Infobox content must
use the same canonical resolution and classification flow as links in prose and
captions. Unsupported or non-playable links retain readable labels without
becoming interactive. Infobox titles and internal section labels do not become
article headings, table-of-contents entries, or same-article jump targets.

Discover Infobox image candidates through the same shared image pipeline used
by ordinary article figures. Apply the standard image acceptance policy exactly,
including its URL, MIME, dimensions, attribution, license, accessibility, and
decorative/interface-media rules. Do not copy or redefine that policy inside
Infobox handling, so later shared-policy improvements apply automatically.
Qualifying static maps are ordinary images. Interactive maps, map controls,
galleries, and embedded applications remain unsupported.

Represent retained audio and video elements with compact, noninteractive typed
placeholders that distinguish the media kind and may use generic wording such as
"Audio element omitted." Placeholders expose no source URL, playback controls,
duration, or upstream markup. Eligible links in surrounding Infobox content
remain Navigation Nodes.

Render the typed block through a dedicated production React Infobox component.
The Article Document renderer delegates Infobox blocks to this component and
passes through the existing Navigation callback. The component owns semantic
Infobox presentation, supported fact/list/media rendering, and accessible local
expand/collapse behavior; it does not parse upstream HTML or revalidate links or
images. It must work anywhere the production Article Document renderer is used,
including the Playable Article Lab and the future Duel screen.

Render the Infobox at its normalized source position and expanded by default.
Provide an accessible Wiki Duel-owned collapse control whose activation is not
a Navigation. Within the reusable Playable Article area, the Infobox should
float beside introductory prose when space permits and become a readable
full-width block at narrow widths. Upstream table dimensions and styling do not
enter the client contract.

**Key interfaces:**

- `ArticleDocument` / `ArticleBlock` — add a dedicated typed Infobox block with
  enough semantic structure for titles, sections, facts, supported content, and
  media placeholders; do not add arbitrary HTML or generic-table payloads.
- Existing Article inline, list, figure, Navigation destination, and image
  attribution contracts — reuse their behavior wherever Infobox content needs
  the same semantics rather than defining parallel forms.
- Candidate discovery and Article Document normalization — traverse recognized
  Infobox structure for eligible links and image candidates, then emit the typed
  block with granular safe fallback and diagnostics.
- Shared image acceptance policy — remain the single authority for whether a
  discovered image becomes retained attributed media.
- `ArticleDocumentRenderer` — dispatch typed Infobox blocks to a dedicated
  production component and forward the existing Navigation callback.
- Dedicated Infobox renderer component — accept the typed Infobox block and
  Navigation callback, render all supported content, and own accessible local
  expansion state and responsive presentation.

**Acceptance criteria:**

- [x] Recognized Wikipedia Infoboxes become dedicated typed Article Document
      blocks at their normalized source positions; no raw or sanitized Infobox
      HTML crosses the server boundary.
- [x] Every supported title, section, fact, line/paragraph group, simple list,
      inline element, image, caption, and media placeholder is retained in
      source order without importance heuristics or arbitrary item limits.
- [x] Eligible Infobox links are resolved and classified through the same
      authoritative Navigation flow as prose and caption links, and activating
      them invokes the surrounding renderer's canonical Navigation callback.
- [x] Unsupported and non-playable links remain readable and noninteractive,
      consistent with existing Article Document behavior.
- [x] Infobox image candidates use the shared discovery/enrichment seam and the
      standard image acceptance policy exactly; Infobox code does not duplicate
      or specialize that policy.
- [x] Qualifying static maps render as ordinary attributed images, while
      interactive maps, galleries, controls, forms, and embedded applications
      are omitted safely.
- [x] Audio and video produce typed, noninteractive omission placeholders with
      no playable media, executable source, or upstream controls.
- [x] Supported descendants survive beside malformed or unsupported content;
      an item disappears only when no meaningful supported content remains, and
      an otherwise valid Playable Article does not fail because one Infobox or
      Infobox item is malformed.
- [x] Infobox-only content does not satisfy the existing meaningful article-body
      normalization requirement.
- [x] Infobox titles and section labels remain internal to the Infobox and do
      not enter the Article Document table of contents or heading-target model.
- [x] A dedicated production Infobox component accepts the typed block and
      Navigation callback, and the Article Document renderer uses it in both the
      Lab and other production-renderer consumers.
- [x] Infoboxes render expanded by default at their source position, provide a
      keyboard-operable and accessibly named collapse control, and collapsing
      does not request or count as Navigation.
- [x] Infoboxes are readable beside introductory prose at wide widths and as
      full-width content at narrow widths without horizontal page overflow.
- [x] Existing omission diagnostics account for unsupported Infobox structures,
      links, and images without exposing upstream HTML or adding diagnostics to
      the production Playable Article contract.
- [x] Focused server fixtures cover representative person and place Infoboxes,
      mixed label/value shapes, simple lists, multiple links and images,
      audio/video, static and interactive maps, malformed nesting, hostile
      markup, granular fallback, and the meaningful-body invariant.
- [x] Focused client tests cover the dedicated component, renderer dispatch,
      semantic content, Navigation callbacks, image and placeholder rendering,
      keyboard collapse behavior, and narrow/wide presentation semantics.
- [x] Existing Article Document normalization, Navigation, image-policy,
      renderer, Playable Article area, and Lab tests continue to pass.

**Out of scope:**

- Generic Wikipedia data-table rendering or a generic table node
- Changes to the shared image discovery or acceptance policy
- Audio or video playback, media URLs, controls, or duration metadata
- Interactive map rendering, map controls, galleries, or embedded applications
- External links as interactive destinations
- Reproducing upstream row spans, column spans, dimensions, classes, or styling
- Infobox importance ranking, semantic fact selection, truncation, or item caps
- Production Duel integration beyond keeping the component reusable there
