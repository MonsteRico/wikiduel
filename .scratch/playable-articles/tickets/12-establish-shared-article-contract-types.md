# Establish Shared Playable Article Contract Types

Status: needs-triage
Scope: MVP required
Category: enhancement

## Parent

- [`playable-articles/01`](./01-fetch-and-sanitize-playable-articles.md)
- [Playable Articles spec](../spec.md)

## Problem

The client currently reaches into `wikiduel-server/src` to reuse Playable Article and Article Document types. Although the import is type-only and adds no runtime server code to the browser bundle, it makes the client depend on the server's private source layout. Duplicating the types in both packages would instead create contract drift and coordinated-edit risk.

## What to build

Establish a neutral, explicit contract boundary for types that legitimately cross between the server and client. Move the Playable Article, Article Document, Navigation Node, revision, and attribution contracts behind that boundary, then migrate both packages to consume it without cross-package source imports or duplicated structural definitions.

Choose and document the repository-standard mechanism during triage. A dedicated shared workspace package is the expected option, but a generated contract may be used if it provides an equally explicit ownership, dependency, build, and drift-detection story. Keep server-only gateway, repository, normalization, failure-handling, and upstream adapter types private to the server.

## Acceptance criteria

- [ ] One neutral contract boundary owns the client/server Playable Article, Article Document, Navigation Node, revision, article attribution, and image attribution types.
- [ ] The client and server consume that boundary through explicit package or generated-contract dependencies rather than importing from each other's source trees.
- [ ] Cross-boundary contract types have one authoritative definition or an automated generation/check step that fails on drift.
- [ ] Server-only repository, gateway, normalizer, configuration, cache, diagnostics, and upstream package types do not enter the shared contract.
- [ ] The client production bundle contains no server runtime code and can build without resolving server implementation modules.
- [ ] Existing server normalization/repository coverage and client Article Document renderer coverage continue to pass without behavioral changes.
- [ ] Root workspace install, test, lint, type-check, and build commands include and validate the chosen contract boundary.
- [ ] The ownership and dependency direction are documented in the relevant ADR or durable architecture documentation.

## Out of scope

- Changing the serialized Playable Article contract or product behavior
- Adding new Article Document node kinds
- Moving server-only implementation code into a shared package
- Replacing the WebSocket message protocol
